### Документ 2: functional-req.md

### ФТ-01: Создание проекта
**Описание:** Система должна предоставлять пользователю возможность создать новый проект, указав название (обязательно) и выбрав цветовую метку из палитры 6–8 цветов. При попытке создания с пустым названием система должна показывать ошибку валидации и не создавать проект.
**Входные данные:** Название проекта (строка), цветовая метка (выбор из палитры)
**Ожидаемый результат:** Новый проект с четырьмя пустыми колонками создан и отображается на боковой панели. Данные сохранены в localStorage.
**Источник:** БТ-01, UC-01, Features

### ФТ-02: Отображение канбан-доски
**Описание:** При открытии проекта задачи должны отображаться в виде канбан-доски с четырьмя колонками: «Бэклог», «В работе», «На проверке», «Готово». Каждая колонка должна иметь шапку с названием и счётчиком задач.
**Входные данные:** Выбор проекта
**Ожидаемый результат:** Отображение задач, распределённых по колонкам согласно их статусу. Счётчики отображают количество задач в каждой колонке.
**Источник:** БТ-02

### ФТ-03: Создание задачи
**Описание:** Система должна предоставлять форму в модальном окне для создания задачи с полями: название (обязательно), описание (необязательно), приоритет (выбор из высокий/средний/низкий, по умолчанию средний), дедлайн (date picker, необязательно), теги (ввод через запятую). При пустом названии кнопка «Создать» не должна создавать задачу, а поле должно подсвечиваться красным.
**Входные данные:** Название, описание, приоритет, дедлайн, теги
**Ожидаемый результат:** Созданная задача добавляется в выбранную колонку (или «Бэклог»), счётчик колонки увеличивается, данные сохраняются в localStorage.
**Источник:** БТ-03, UC-02, Features

### ФТ-04: Перемещение задачи между колонками
**Описание:** Система должна предоставлять на карточке задачи кнопки «→» и «←» для перемещения задачи между колонками. Кнопка «→» перемещает задачу вперёд (Бэклог → В работе → На проверке → Готово), кнопка «←» — назад. Кнопки должны быть неактивными или скрытыми, если перемещение в данном направлении невозможно (задача в первой или последней колонке).
**Входные данные:** Нажатие на кнопку «→» или «←»
**Ожидаемый результат:** Задача перемещается в соседнюю колонку, счётчики обеих колонок обновляются, данные сохраняются в localStorage.
**Источник:** БТ-04, UC-03, Features

### ФТ-05: Визуальное выделение просроченных задач
**Описание:** Система должна визуально выделять задачи, срок дедлайна которых уже истёк, путём отображения красной метки «Просрочено». Выделение должно основываться на текущей дате и дате дедлайна задачи.
**Входные данные:** Дата дедлайна задачи, текущая дата
**Ожидаемый результат:** На карточке задачи отображается красная метка «Просрочено».
**Источник:** БТ-05

### ФТ-06: Визуальное выделение задач с дедлайном сегодня
**Описание:** Система должна визуально выделять задачи, срок дедлайна которых совпадает с текущей датой, путём отображения жёлтой метки «Сегодня».
**Входные данные:** Дата дедлайна задачи, текущая дата
**Ожидаемый результат:** На карточке задачи отображается жёлтая метка «Сегодня».
**Источник:** БТ-06

### ФТ-07: Редактирование задачи
**Описание:** Система должна предоставлять возможность редактировать задачу через модальное окно, предварительно заполненное текущими данными. Пользователь может изменять любые поля (название, описание, приоритет, дедлайн, теги) и сохранить изменения.
**Входные данные:** Изменённые поля задачи
**Ожидаемый результат:** Карточка задачи обновляется, изменения сохраняются в localStorage.
**Источник:** БТ-07, UC-04

### ФТ-08: Удаление задачи
**Описание:** Система должна предоставлять возможность удаления задачи с предварительным подтверждением операции. После подтверждения задача удаляется, счётчик колонки уменьшается.
**Входные данные:** Нажатие на кнопку «✕», подтверждение в диалоговом окне
**Ожидаемый результат:** Задача удалена из колонки, данные сохранены в localStorage.
**Источник:** БТ-07, UC-05

### ФТ-09: Переключение между представлениями
**Описание:** Система должна предоставлять кнопки в шапке доски для переключения между режимами канбан-доски и списка (табличного представления). Табличное представление должно содержать колонки: «Название», «Приоритет», «Статус», «Дедлайн», «Теги».
**Входные данные:** Нажатие кнопки «Список» или «Доска»
**Ожидаемый результат:** Представление переключается. Активные фильтры и поиск сохраняются.
**Источник:** БТ-08, UC-07, Features

### ФТ-10: Фильтрация задач
**Описание:** Система должна предоставлять фильтры по приоритету (Все/Высокий/Средний/Низкий), тегу (выпадающий список всех тегов проекта) и статусу колонки. Фильтры должны применяться с AND-логикой. Должен отображаться счётчик активных фильтров и кнопка «Сбросить фильтры».
**Входные данные:** Выбор значений из выпадающих списков приоритета, тега, статуса
**Ожидаемый результат:** Отображаются только задачи, удовлетворяющие всем выбранным фильтрам.
**Источник:** БТ-09, UC-06, Features

### ФТ-11: Поиск задач в реальном времени
**Описание:** Система должна предоставлять поле поиска, скрывающее карточки задач, не содержащие введённый текст в названии или описании. Поиск должен выполняться регистронезависимо и немедленно при вводе текста.
**Входные данные:** Текст поиска (строка)
**Ожидаемый результат:** Отображаются только задачи, название или описание которых содержит введённый текст.
**Источник:** БТ-10, UC-06, Features

### ФТ-12: Отображение дашборда со статистикой
**Описание:** Система должна отображать на дашборде карточки-метрики с общим числом задач, распределением по статусам (В работе, На проверке, Готово), числом просроченных задач. Для каждого проекта должна отображаться полоса прогресса в процентах задач в колонке «Готово». Также должен быть список задач с дедлайном в ближайшие 7 дней.
**Входные данные:** Переход на дашборд
**Ожидаемый результат:** Отображение статистики по всем проектам с возможностью просмотра деталей.
**Источник:** БТ-11, UC-08, Features

### ФТ-13: Сохранение и восстановление данных
**Описание:** Система должна автоматически сохранять все данные в localStorage при каждом изменении (создание, редактирование, удаление, перемещение задач и проектов). При загрузке приложения данные должны восстанавливаться из localStorage. При первом запуске должен создаваться демо-проект.
**Входные данные:** Действия пользователя по изменению данных
**Ожидаемый результат:** Данные сохранены в localStorage под ключом `taskflow_data` и могут быть восстановлены при перезагрузке.
**Источник:** БТ-12, Features

### ФТ-14: Обновление счётчиков в реальном времени
**Описание:** Система должна обновлять счётчики задач в шапках колонок немедленно при добавлении, удалении или перемещении задач. Счётчики на дашборде также должны обновляться в реальном времени.
**Входные данные:** Любое изменение количества задач в колонке
**Ожидаемый результат:** Счётчики отображают актуальное количество задач.
**Источник:** БТ-14
